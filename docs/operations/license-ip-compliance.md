# License And IP Compliance

## Purpose

This runbook governs third-party runtime and CI dependencies distributed or used by
`lotus-idea`. It provides deterministic engineering evidence for review; it does not replace
bank legal advice, procurement approval, vendor terms, or organization-level security policy.

## Control Model

| Artifact | Authority | Enforcement |
| --- | --- | --- |
| `LICENSE` and `pyproject.toml` | Service-code license | Package metadata and repository review |
| `contracts/compliance/lotus-idea-license-policy.v1.json` | Component classification, obligations, exceptions, and approval roles | `scripts/license_compliance_policy.py` through `make license-compliance-gate` |
| Runtime and CI resolved locks | Exact dependency closure | SHA-256 and component/version/scope reconciliation |
| `THIRD_PARTY_NOTICES.md` | Reproducible attribution view | Generated byte-for-byte from the policy |
| Release manifest and CycloneDX SBOM | Published-image linkage | `make license-release-evidence-gate` |
| `.github/CODEOWNERS` | Repository review routing | GitHub branch-protection review configuration |

The service is proprietary (`LicenseRef-Lotus-Proprietary`). Approved licenses pass directly.
Conditional licenses pass only with explicit obligations. Denied, unknown, malformed, unlocked,
or unclassified components fail closed unless a current exception satisfies every approval rule.

## Current Conditional Components

| Component | License | Required posture |
| --- | --- | --- |
| `certifi` | `MPL-2.0` | Preserve notices; make modified MPL-covered files available under MPL terms. |
| `psycopg`, `psycopg-binary` | `LGPL-3.0-only` | Preserve notices, retain a relink/replace mechanism, and provide the corresponding library-source offer. |

The Python base image requires release-time container package inventory. Trivy scans that image,
but a vulnerability scan is not legal clearance. The policy separately declares generated code,
model assets, and data assets; no externally sourced asset in those categories is currently
distributed by this repository.

## Dependency Change Procedure

1. Change root dependency pins and run `make dependency-refresh`.
2. Add or update every resolved component in the policy with exact version, scope, SPDX
   expression, source URL, attribution status, and conditional obligations.
3. Update the runtime and CI lock hashes in the policy.
4. Regenerate notices with
   `.venv/Scripts/python.exe scripts/license_compliance_gate.py --write-notice` on Windows or the
   equivalent repository Python on other platforms.
5. Run `make license-compliance-gate`, focused mutation tests, and the repository-native lane.
6. Obtain application-owner, security, and legal review. `CODEOWNERS` routes repository review;
   it is not proof of the separate legal and security decisions.

Never edit `THIRD_PARTY_NOTICES.md` independently. Never suppress an unknown or denied result by
changing an SPDX expression to a more convenient value.

## Exception Procedure

An exception must include a unique `exception_id`, exact component and SPDX expression, bounded
reason, expiry date, all three required approval roles, and an immutable review or ticket reference.
Expired, incomplete, or evidence-free exceptions fail closed. Renewals require a new review before
the prior expiry; permanent exceptions are prohibited.

Escalate suspected licensing ambiguity, conflicting upstream metadata, copyleft/network-use
exposure, dual-license selection, generated-code provenance, model/data usage rights, or base-image
package uncertainty to the service owner, security, and legal reviewers before merge or release.
Do not put confidential legal advice or proprietary agreements in public issues or build artifacts.

## Release Evidence

Main Releasability is the only image publication path. It records the policy version, both lock
hashes, NOTICE digest, SBOM serial number, active exception IDs, and digest-pinned target artifact
in `release-evidence.json`. The release gate verifies those values against the exact policy, SBOM,
and final image digest. `LICENSE`, notices, and the policy accompany release evidence, and the image
contains the service license and notices.

This proves deterministic artifact linkage. It does not prove organization approval unless the
required immutable reviews exist, and it does not certify Debian package licensing merely because
the application dependency inventory passes.

## Failure Triage

| Failure | Required action |
| --- | --- |
| Lock hash or inventory drift | Reconcile the complete resolved closure; do not update only the hash. |
| Unknown or denied SPDX result | Verify upstream authority, classify correctly, replace the component, or obtain a bounded exception. |
| Conditional obligations missing | Record actionable obligations and confirm release/distribution handling. |
| NOTICE mismatch | Regenerate from the reviewed policy and inspect the diff. |
| Release evidence mismatch | Regenerate from the exact final digest/SBOM; never patch evidence by hand. |
| Base-image uncertainty | Stop release and escalate to platform container policy, security, and legal owners. |

## Validation

```powershell
make license-compliance-gate
.\.venv\Scripts\python.exe -m pytest tests/unit/test_license_compliance_gate.py -q
make license-release-evidence-gate RELEASE_MANIFEST=<manifest> RELEASE_SBOM=<sbom>
```

The last command requires generated mainline release artifacts. Local policy success cannot be
used as evidence that an image was published, signed, attested, legally approved, or promoted.
